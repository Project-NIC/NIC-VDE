```
   контейнер NIC-MLA ──▶ декод схемы ──▶ строки ──▶ CSV / SQLite
```

> **Сначала прочти это.** Это близнец **NIC-GLUE-IN**, та же идея, развёрнутая
> в обратную сторону: GLUE-IN заводит строку данных *в* контейнер; GLUE-OUT
> проходит готовый контейнер *обратно наружу* в таблицу. Как и у близнеца, это
> **рабочий пример плюс маленький каталог опций**, а не фреймворк. Непреходящая
> ценность снова — **[справочник выравнивания библиотек](#справочник-выравнивания-библиотек)**:
> швы, прочитанные с другого конца. Пример делает самое простое полезное:
> открыть контейнер, декодировать каждую RAW-запись через её самоописывающую
> схему и выгрузить всё в **CSV или SQLite**. **Сжатие NIC-DMD эта читалка
> распаковывает автоматически**; вне области остаётся лишь шифрование (NIC-KSF)
> — см. [ниже](#использование-nic-dmd-и-nic-ksf). **NIC-VDE** —
> интерактивный просмотрщик тех же файлов; GLUE-OUT — безголовый путь экспорта.

## Справочник выравнивания библиотек

Библиотеки NIC намеренно *тупые и независимые*: MLA хранит непрозрачные байты,
VDE просматривает файлы, KSF преобразует байты, DMD кодирует пакеты. Ни одна не
знает об остальных. Клей — это любой код, который выравнивает эти швы. GLUE-IN
их *записывает*; вся работа читалки — *прочитать те же швы обратно*. Те, что
использует эта простая читалка:

| Шов | Что несёт контейнер | Что с этим делает читалка |
|---|---|---|
| **Вид записи** | MLA не несёт байта типа; у записи есть только бит `compressed` + `kf_back` (0 = keyframe). Файлы однородны — типа TEXT/EVENT/CLASS больше нет, смысл задаёт СХЕМА | выводит *вид* `raw` / `keyframe` / `delta` из них, затем декодирует в именованные значения: raw прямо по схеме, сжатые через DMD (см. ниже); строка пуста только без совпадения со схемой |
| **Станция** | MLA-лог хранит 1-байтовый *индекс* станции; реальные номера живут в таблице станций в префиксе | переводит индекс → регион/номер, чтобы экспортные строки несли реальные номера |
| **Время** | у MLA-лога есть выделенный 4-байтовый `timestamp`; поле схемы `log("datetime")` его описывает | время берётся прямо из заголовка лога — никогда не выкапывается из блока данных |
| **Раскладка полей** | схема делит поля `log(...)` (заголовок) и поля `data(...)` (нагрузка) | `mla_decode_payload` разбивает упакованный блок обратно на именованные масштабированные значения |
| **Целостность** | MLA покрывает лог-запись (и опц. блок данных) через CRC16 | слоты с плохим CRC ядро MLA пропускает при монтировании — читалка видит только зафиксированные записи |

Если файл уважает эту таблицу на входе (любой GLUE-IN — да), он читается прямо
обратно здесь и в NIC-VDE, независимо от того, как сделано остальное.

## Что даёт пример

Намеренно маленькая читалка / экспортёр над одним MLA-контейнером:

- **`GlueReader`** — открой контейнер, затем итерируй его или экспортируй. Он
  читает самоописывающие таблицы схемы/станций из префикса, декодирует каждую
  RAW-нагрузку измерения в именованные масштабированные значения, переводит
  индекс станции в реальный регион/номер и сериализует всё в **CSV** или
  **SQLite**.

```python
from nic_glue_out import GlueReader

with GlueReader("weather.mla") as r:
    for rec in r:                          # декодированные записи, от старых (raw + сжатые)
        if rec.values is not None:         # декодированные значения (raw или распакованные из DMD)
            print(rec.timestamp, rec.station_label,
                  {n: v for n, _u, v in rec.values})
        else:                              # нет соответствия схеме — показать сырые байты
            print(rec.timestamp, rec.kind, rec.block.hex())

    r.write_csv("weather.csv")             # → idx,time,unix,sta_idx,region,number,kind,length,<поля…>
    r.write_sqlite("weather.db")           # → база SQLite из одной таблицы
```

```bash
python3 examples/weather_export.py          # строит образец, затем выгружает weather.csv + weather.db
python3 tests/test_glue.py                  # или: pytest tests/
```

## Опции дизайна & как сделать

Это *возможности*, не требования — выбери, что подходит. Пример реализует самое
простое полезное чтение + экспорт; остальное — короткий список ручек.

### 1. Цели экспорта — CSV или SQLite

Оба выпадают из одних и тех же собранных строк; модуль `export` — тупой
сериализатор (о MLA ничего не знает). `to_csv()` возвращает UTF-8 байты;
`to_sqlite()` возвращает базу из одной таблицы в виде байтов. Передай `raw=True`,
чтобы оставить целые числа «с провода» вместо масштабированных физических
значений. Добавь свою цель (Parquet, JSON Lines, сокет), написав ещё один
`to_…`, который потребляет те же колонки `(name, sql_decl)` + кортежи строк.

### 2. Откуда берётся метка времени

Читалка время никогда не угадывает: у MLA-лог-записи есть выделенный 4-байтовый
`timestamp`, отдельный от блока данных, а поле схемы `log("datetime")` его лишь
*описывает*. Поэтому читалка берёт его прямо из заголовка лога (`rec.timestamp`)
— блок данных это чистая нагрузка сенсоров. Точная инверсия шва GLUE-IN «куда
идёт метка времени»: время живёт в заголовке, никогда не дублируется в данных, и
на входе, *и* на выходе.

### 3. Фильтрация

Читалка грузит весь контейнер в RAM (документированная хост-модель) и фильтрует
на стороне хоста: `records(station=…, time_from=…, time_to=…)`. Никакого индекса
на диске нет — это линейный проход, тот же результат, что фильтровать каждую
запись.

### 4. Файлы без схемы

Контейнер, записанный без схемы, тоже читается: каждая запись сводится к одной
колонке `value` (текст как текст, маленькие нагрузки как целое, иначе hex). Файл
*со* схемой получает по одной именованной колонке на поле данных.

### Использование NIC-DMD и NIC-KSF

- **NIC-DMD (встроено).** Если писатель использовал сжатый канал GLUE-IN, такие
  записи несут бит `compressed` (вид `keyframe` / `delta`). Эта читалка
  **распаковывает их автоматически**: проигрывает поток каждой станции по порядку
  через per-station `DmdDecoder(ширина)` (`ширина` = суммарная ширина данных
  схемы), а результат затем прогоняет через тот же декод схемы, что и RAW-путь —
  так что сжатые и RAW-строки экспортируются одинаково. Строка показывает пустые
  ячейки только если для неё вообще нет совпадения со схемой. **NIC-VDE** такие
  файлы тоже просматривает.
- **NIC-KSF (шифрование).** KSF живёт на *транспортном* пути, никогда в хранении:
  отправитель шифрует перед передачей, получатель расшифровывает *до* того, как
  байты сохранены — так что контейнер держит открытый текст и этой читалке ключ
  не нужен. Добавь его на стороне приёма (`приём → ksf_decrypt → … → сохрани`),
  зеркало отправляющей стороны GLUE-IN. См. NIC-GLUE-IN.

## Раскладка

```
nic_glue_out/       сам клей (GlueReader) + тупой CSV/SQLite экспортёр
examples/           запускаемая weather читалка / экспортёр
tests/              тесты обратного чтения + декода + экспорта
third_party/        вендоренная копия NIC-MLA (см. VENDORED.md)
tools/              sync_vendor.py — пересинхронизация third_party/ из канонических NIC-MLA/NIC-DMD
```

Чистый Python 3.10+, без внешних пакетов — зависимость вендорена, экспортёр —
stdlib `sqlite3`.
