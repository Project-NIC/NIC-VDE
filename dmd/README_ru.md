# NIC DMD — Delta Markov Duda

## Протокол сжатия данных для встроенных устройств

[![License: MIT](https://img.shields.io/badge/License-MIT-red.svg)](https://opensource.org/licenses/MIT)

---

## Что такое DMD?

DMD — это кроссплатформенный протокол сжатия данных для небольших пакетов информации с метеостанций, счётчиков электроэнергии, GPS-трекеров и других встроенных устройств. Он разработан для передачи по технологиям с ограниченной полосой пропускания, таким как LoRa.

Протокол полностью функционален на контроллере ATmega328 и не требует больших словарей или таблиц поиска в памяти. Каждый пакет сжимается независимо — с помощью адаптивного выбора оптимального метода из пяти кандидатов.

---

## Почему DMD?

Существующие библиотеки сжатия данных для встроенных устройств либо требуют сотни дополнительных байт ОЗУ (Heatshrink), либо нужно передавать таблицу Хаффмана вместе с данными. DMD выбирает другой путь — комбинирует несколько простых методов с эвристическим анализом и выбирает оптимальный результат для каждого пакета отдельно.

**Основные преимущества:**
- Фиксированная таблица Хаффмана только в ПЗУ (64B), без дополнительной ОЗУ
- Адаптивный выбор метода для каждого пакета — до 5 кандидатов
- Полностью детерминированная распаковка — без потери данных
- Максимальное увеличение объёма данных на 1 байт (заголовок) в худшем случае
- Реализация на Python и C (ATmega328 / Arduino)

---

## Когда DMD неэффективен

DMD разработан для данных, которые изменяются медленно и предсказуемо — значения сенсоров, GPS-координаты, промышленная телеметрия. Если входные данные случайные, зашифрованные или уже сжатые, DMD добавит только 1 байт заголовка и отправит их как RAW. Это правильное поведение — без потерь, без ухудшения.

---

## Совместимость

**Python:** 3.10 и новее (используются типовые аннотации `bytes | None`).

**C:** C99 и новее. Протестировано с GCC на ПК (Linux/Windows) и AVR-GCC для ATmega328. Без зависимостей от стандартной библиотеки, кроме `<string.h>`. Внутренние буферы размеруются с помощью C99 VLA в соответствии с фактической длиной пакета.

**Arduino:** Скопируйте `c/nic_dmd.c` и `c/nic_dmd.h` в папку проекта. Совместимо с Arduino IDE 1.8+ и 2.x (AVR-GCC поддерживает C99 VLA).

**Примечание для других компиляторов:** IAR, Keil и MSVC C++ не поддерживают VLA. Для этих наборов инструментов можно определить `-DDMD_PKT_MAX_BUILD=N` при компиляции (например, 32 или 64), и буферы будут фиксированного размера.

**Зависимости для fetch/benchmark:** `pip install requests`

**Длина пакета:** Минимальное техническое ограничение — 1B, но при размере менее 16B сжатие практически неэффективно — накладные расходы заголовка (1B) и состояния ANS (2B) поглощают большую часть потенциальной экономии. Рекомендуемый минимум — **16B**. Максимум — **255B**. Для передачи LoRa практический предел нагрузки — 51–64B в зависимости от коэффициента расширения и региона. DMD достигает лучших результатов на данных, где соседние пакеты изменяются медленно — обычно 16–64B телеметрия сенсоров.

## Валидация и целостность данных

В целях достижения максимальной производительности и абсолютной минимизации нагрузки на процессор библиотека не выполняет никаких дополнительных проверок заголовка и не валидирует длину переданных данных.

Конструкция протокола строго предполагает, что проверка целостности (например, аппаратное CRC) и отброс повреждённых или пустых пакетов обрабатываются нижним уровнем транспорта или основной программой (обычно самим радиомодулем, логикой сбора данных и т.д.). Пользователь библиотеки должен убедиться на прикладном уровне, что в функции сжатия и распаковки поступают только структурно корректные данные. Делегированием этой ответственности удалось добиться низких расходов памяти без растраты циклов процессора.

## Методы сжатия

**1. Кодирование дельты + ZigZag (DELTA1)**

Кодировщик вычитает каждое значение из предыдущего (формируя разности). Декодировщик складывает их обратно — полностью обратимо, без потерь. ZigZag преобразует целые числа со знаком (положительные и отрицательные) в маленькие беззнаковые целые числа для лучшей энтропии. Этот метод доминирует примерно в 70% тестовых случаев, потому что данные с естественных датчиков изменяются медленно.

**2. FLAG — исключение нулей**

Заменяет каждую последовательность нулевых байт битом в битовой карте. Нагрузка: `[1B длина][битовая карта][ненулевые байты]`. Тривиально декодируется — без плавающей запятой, без состояния. Включается, если >= 30% байт нулевые. Очень эффективно на разреженных данных (много нулей, чередующихся с реальными значениями).

**3. ANS — асимметричные системы счисления**

Арифметическое кодирование для байт с кодами переменной длины. Кодировщик строит таблицу частот на лету и кодирует байты как поток. Декодировщик использует обратную конечный автомат — словарь не нужен. Быстро на маленьких пакетах, особенно на текстовых и CSV данных.

Нагрузка ANS содержит длину данных (1B), состояние (2B — uint16_t) и закодированные байты. Активируется только если доля нулевых байт >= 45% (эвристика). Оба кодировщик и декодировщик имеют ранний выход — если результат превышает лимит, вычисление тут же останавливается.

**4. Nibble Хаффман (HUF)**

Фиксированная таблица Хаффмана, обученная на комбинированных метеоданных и GPS данных после дельты+ZigZag. Кодирует каждый байт как два кода nibble (hi и lo). Таблица хранится в ПЗУ (32B PROGMEM на ATmega), никакого дополнительного ОЗУ.

Максимальная длина кода — 6 бит, в среднем ~3.2 бита на байт. Побеждает особенно на IoT, промышленных и сложных данных, где нули редки, но распределение nibble соответствует таблице.

**5. Комбинация FLAG+HUF**

FLAG сначала удаляет нулевые байты в битовую карту, затем Хаффман сжимает оставшиеся ненулевые байты. Нагрузка: `[1B длина][битовая карта][1B действительные биты HUF][HUF поток]`. Лучшее из обоих миров — детерминированное исключение нулей + энтропийное сжатие остатка.

**Keyframe и стартовый кадр**

Образец с номером 0 — это keyframe. Поскольку нет предыдущего пакета для вычисления дельты, метод разности и ZigZag пропускаются. Данные обрабатываются напрямую методами FLAG, HUF, FLAG+HUF или ANS. Keyframe происходит автоматически каждые 7 пакетов или после сброса устройства.

## Использование

### Python

```python
from nic_dmd import DmdEncoder, DmdDecoder

PKT_LEN = 16
enc = DmdEncoder(PKT_LEN)
dec = DmdDecoder(PKT_LEN)

data = bytes([0xFC, 0x18, 0x21, 0x34, 0x01, 0x81,
              0x04, 0xCE, 0x00, 0x00, 0xFC, 0x7C,
              0xFC, 0xA8, 0x00, 0x00])

compressed   = enc.compress(data)
decompressed = dec.decompress(compressed)

print(f"Сжато: {PKT_LEN}B → {len(compressed)}B")
assert decompressed == data
```

### C (ATmega328 / Arduino)

```c
#include "nic_dmd.h"

dmd_encoder_t enc;
dmd_decoder_t dec;

void setup() {
    dmd_encoder_init(&enc, 16);   // длина пакета — должна совпадать с обеих сторон
    dmd_decoder_init(&dec, 16);
}

void loop() {
    uint8_t data[16]          = { /* данные сенсора */ };
    uint8_t compressed[DMD_OUT_MAX];   // DMD_OUT_MAX = длина пакета + 1 (до 256B)
    uint8_t decompressed[16];

    uint16_t comp_len = dmd_compress(&enc, data, compressed);
    lora.send(compressed, comp_len);

    // На приёмнике:
    int res = dmd_decompress(&dec, compressed, comp_len, decompressed);
    if (res != 0) {
        // res < 0 → пакет повреждён, см. таблицу возвращаемых значений ниже
    }
}
```

### Возвращаемые значения и коды ошибок

Каждая функция возвращает число при завершении. Это число — единственный способ, которым библиотека вам что-то сообщает — никаких выводов, никакого логирования (для экономии памяти и производительности). Вышестоящая программа (та, что использует библиотеку) должна прочитать это число и действовать в соответствии с ним.

**`dmd_compress(...)` — сжатие**

Возвращает **длину вывода в байтах** (тип `uint16_t`, т.е. 16-битное число):

| Возвращённое значение | Значение | Что делать |
|---|---|---|
| 2 до 256 | Количество байт для отправки (1B заголовок + сжатые данные) | Отправьте ровно столько байт из `output` |

Сжатие **никогда не ошибается** и не имеет кода ошибки — вы всегда получаете действительную длину. В худшем случае (пакет 255B, который не сжимается, например, случайные или зашифрованные данные), результат **256 B**, т.е. на 1 байт больше входа. Это называется максимальное расширение на 1B (этот 1 байт — обязательный заголовок). Поэтому возвращаемый тип 16-битный — чтобы вместить число 256. Выходной буфер должен иметь размер `DMD_OUT_MAX` (= длина пакета + 1).

**`dmd_decompress(...)` — распаковка**

Возвращает **код состояния** (тип `int`). Распакованные данные найдены в `output` только когда код `0`:

| Возвращённое значение | Значение | Что делать |
|---|---|---|
| `0` | OK — всё прошло нормально, `output` содержит исходные данные | Используйте `output` |
| `-1` | Повреждённый или неверный ввод (пустой пакет или длина нагрузки не совпадает) | Отбросьте пакет, данные непригодны |
| `-3` | Зарезервированная версия протокола (в заголовке `sample_num = 7`) | Этот пакет не принадлежит этой версии библиотеки — отбросьте его |

Отрицательное число всегда означает "что-то не так, не используйте данные". Сама библиотека не выполняет проверки целостности (CRC и т.д.) — она предполагает, что поврежденные пакеты будут перехвачены нижним уровнем (радиомодулем). Коды `-1` и `-3` — это просто последняя страховка от явно бессмысленного ввода.

> **Версия Python ведёт себя идентично.** `DmdEncoder.compress()` возвращает такой же длины вывод (включая те 256 B в худшем случае) и `DmdDecoder.decompress()` при повреждённом или неверном вводе вызывает **исключение** вместо отрицательного кода — это эквивалент ошибки C в Python (зарезервированная версия протокола конкретно вызывает `ValueError`). Обработайте её через `try/except`. При всём остальном одинаковый ввод производит **байтово идентичный вывод** в C и Python, поэтому вы можете сжимать на устройстве C и распаковывать на сервере/Raspberry Pi на Python (и наоборот).

### Компиляция для других компиляторов (без VLA)

Если ваш компилятор не поддерживает C99 VLA (IAR, Keil, MSVC C++), определите максимальную длину пакета при компиляции:

```
gcc -DDMD_PKT_MAX_BUILD=32 c/nic_dmd.c ...
```

Буферы будут скомпилированы на фиксированный размер 32B. Для проектов с одной фиксированной длиной пакета (типичный случай Arduino) этот вариант идеален.

```
makefile  Сборка C-библиотеки и запуск тестов C и Python
```

| Файл                    | Описание                                             |
| ----------------------- | ---------------------------------------------------- |
| `python/nic_dmd.py`     | Реализация Python — справочная, для тестирования     |
| `c/nic_dmd.c`           | Реализация C для ATmega328                           |
| `c/nic_dmd.h`           | Файл заголовка                                       |
| `bench/nic_dmd_utils.py`| Вспомогательные функции — анализ и вывод результатов |
| `makefile`              | Компиляция и тестирование                            |
