# NIC-MLA — формат датологгера (profile-ref)

> **Статус:** реализовано (эталон на Python + 33 теста, C-ридер + декодер значений).
> **Аддитивно** к формату v1.2 (одна схема) — 16-байтовая log-запись не меняется и
> файлы v1.2 продолжают работать. Референс: `tools/mla_datalogger.py`, тесты
> `tools/mla_datalogger_test.py`, C `c/datalogger_test.c` + `c/mla_decode.c`.
> Дата: 2026-06-06

## Зачем
Датологгер / LoRa-ретранслятор принимает от **нескольких типов станций** (метео,
электричество, улей, …) и должен писать их в **один** `.mla`. v1.2 несёт одну схему
на файл (один макет + много идентичностей станций); формат датологгера позволяет
**каждой станции нести свой макет столбцов**, при этом разделяя макет, когда
станции одинаковы.

## Модель — profile-ref
- **ПРОФИЛЬ** = макет столбцов (свои дескрипторы полей данных).
- **СТАНЦИЯ** = 8-байтовая непрозрачная идентичность + 1-байтовая ссылка на профиль.
- 1-байтовый **индекс станции** в log-записи выбирает станцию →
  `{ идентичность, ссылка на профиль }` → профиль → декод полезной нагрузки.

```
log.index → СТАНЦИЯ (идентичность + ссылка) → ПРОФИЛЬ (макет) → значения
```

Макет разделяется между одинаковыми станциями (8 метеостанций → 1 профиль + 8
строк станций), но при этом в одном файле допустимы *разные* макеты (метео +
электричество).

## Двоичная раскладка (в префиксе, в слоте `schema_table`, после 34-байт заголовка)
Каждая секция помечена тегом и самоизмерима; читатель проходит их по порядку. Весь
блок покрыт CRC префикса, ровно как таблица схемы v1.2.

```
LOG       : 0x4C  n_log         n_log × 14B дескриптор     (описывает фиксированную 16B запись)
ПРОФИЛИ   : 0x50  n_профилей    [ n_data(1B)  n_data × 14B ] × n_профилей
СТАНЦИИ   : 0x54  n_станций     [ идентичность(8B)  ссылка(1B) ] × n_станций
```

14-байтовый дескриптор поля и `физическое = (raw + offset) × 10^exp10` **те же**,
что в v1.2 (`width 1/2/4 · unit · exp10 i8 · flags · offset i16 · name 8B`). Теги
(0x4C/0x50/0x54) отличаются от тега схемы v1.2 (0x01), поэтому ядро
(`_schema_byte_len` в `nic_mla.py` и `mla_datalogger_size` в `nic_mla.c`)
вычисляет длину обоих форматов — `MlaCore` просто несёт байты.

## Идентичность станции (8 B, непрозрачная)
MLA не придаёт ей смысла; это делает glue. Кодировщики в билдере:
- `dl_gps(lat, lon)` — 2× i32 (градусы ×10⁷, ~1 см)
- `dl_ident(number, region, kind, reserved)` — иерархическая (4× u16)
- `dl_raw(8 байт)` — что угодно
> Четыре электросчётчика в одной коробке → 4 станции с **одинаковым GPS, разным
> `number`, одной ссылкой на профиль** (макет хранится один раз).

## Использование
```python
from mla_schema import MlaField
from mla_datalogger import DataloggerBuilder, DataloggerTables, dl_gps, export_csv
from nic_mla import MlaCore, MlaPosixHAL

# 1) опиши профили + станции
b = DataloggerBuilder()
b.log("datetime")
meteo = b.profile([MlaField("temp", 2, "degC", -2, signed=True),
                   MlaField("hum",  2, "pct",  -1)])
elec  = b.profile([MlaField("power", 2, "W"), MlaField("energy", 4, "kWh")])
b.station(dl_gps(50.0875, 14.4213), meteo)   # станция 1
b.station(dl_gps(49.1951, 16.6068), meteo)   # станция 2 (тот же макет)
b.station(dl_gps(50.0875, 14.4213), elec)    # станция 3 (другой макет)
blob = b.serialize()

# 2) запиши реальный .mla (таблицы едут в слоте schema_table)
hal = MlaPosixHAL.create("weather.mla", 64 * 1024)
with hal:
    m = MlaCore(hal); m.format(file_size=64 * 1024, schema_table=blob)
    t = DataloggerTables.parse(blob)
    m.append(1700000000, station=1, data=t.encode(1, {"temp": 25.45, "hum": 60.0}))
    m.append(1700000060, station=3, data=t.encode(3, {"power": 1500, "energy": 12345}))

# 3) экспорт → одна CSV / SQL-таблица на профиль
export_csv("weather.mla", "out/")
```

## Ограничения
- ≤ 255 профилей, ≤ 255 станций (1-байтовые счётчики / индекс), ≤ 255 столбцов на профиль.
- Нагрузка на запись ≤ 65535 B (log `length` — u16); DMD-сжатая строка ≤ 255 B.
- Файл — это либо v1.2-схема, **либо** датологгер; различаются по тегу на смещении 34.
  Контейнер, CRC, crash-safety, ротация и сжатие идентичны v1.2.

## Что это связывает
- `tools/mla_datalogger.py` — билдер, читатель, декод по профилю, кодировщики идентичности, CSV/SQL экспорт.
- `nic_mla.py` / `c/nic_mla.c` — определение длины блока датологгера (аддитивно).
- `c/mla_decode.c` — C-декодер значений, байт-идентичный Python (и для v1.2).
- Смешение профилей проверено end-to-end на реальном `.mla` в тестах (Python и C).
