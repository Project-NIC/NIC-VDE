# NIC-MLA — Спецификация дизайна формата

> **Статус:** v1.1 · **Дата:** 2026-06-06
> **MLA** = *Matroshka Logging Archive* — универсальный однофайловый контейнер
> (данные + логи в одном файле, как Matroska / tar / DriveSpace).
>
> Этот документ определяет формат v1.1. **Статус реализации:** Python эталон
> (`nic_mla.py`, `nic_mla_archive.py`, `tools/mla_schema.py`) и C библиотеки
> (`c/` — только запись для ATmega + полная для ARM/ПК) готовы и бинарно
> идентичны (проверено кросс-тестом C↔Python).
>
> **Принцип дизайна — тупой контейнер.** MLA только хранит байты: запись лога
> 16 Б под CRC + блок данных, плюс две самоописывающие таблицы в префиксе
> (имена/единицы полей и индекс станции → реальный номер). Вся логика — сжатие,
> шифрование, перевод номеров станций, транспорт — в отдельном слое-клее.

---

## 1. Назначение и область применения

NIC-MLA — это **универсальный контейнер для записи данных** с измерительных станций
(метеостанция, счётчик электроэнергии и т.д.). Цель — один портативный файл, который содержит
**данные и логи вместе** и читается на всех платформах.

**Зачем это нужно:** конец путанице с миллионом форматов. Вместо груд файлов и
инструментов → **одна таблица**, в которую можно «вставить» всё что угодно. Вытащишь карту из
устройства, воткнёшь в компьютер, и **один простой просмотрщик** разберёт структуру
из внутренних регистров, которые так хорошо описаны, что это поймёт даже ребёнок.
Цель не быть «крутым» — цель **простой, интуитивный и тривиальный процесс**,
который сэкономит время и деньги. (То, что части этого подхода есть и в других местах — неважно —
ценность в том, чтобы иметь их **все вместе и с самоописанием**.)

### Целевые платформы и роли

| Платформа | Роль | Что делает |
|---|---|---|
| **ATmega328** (8-бит) | **ТОЛЬКО ЗАПИСЬ** | только добавляет записи (append), примерно каждые 15 минут; нет поиска или редактирования на чипе |
| Arduino 32/64-бит, STM, ESP | запись + опционально чтение | как ATmega + локальное чтение |
| **Хост** (ПК / Raspberry) | чтение, поиск, редактирование | загружает весь логи в ОЗУ, фильтрует, экспортирует |

**Ключевой принцип:** запись тривиальна и надёжна (из-за ATmega), а вся интеллектуальная работа 
(поиск, запросы, редактирование) происходит на хосте, где логи загружаются в ОЗУ сразу.
**На диске нет никакого дерева/AVL — только плоский логи**, который хост сканирует последовательно.
Поля логов разработаны так, чтобы это фильтрование было быстрым (время, индекс станции, тип).

### Вне области применения

- **Редактирование** записей → отдельный проект просмотрщика *Volkov Data Ecosystem* (NIC-VDE).
- **Сжатие** → опционально, выполняется **отдельным методом**; контейнер только **помечает** сжатые
  данные (один бит `compressed` + расстояние `kf_back`), сам сжатие не определяет (см. §4).
- **Прямой сырой доступ к SPI-NOR/NAND** → **экспериментально и заморожено** (см.
  `experimental/`). Целевой носитель — **SD/флеш-карты** — собственный контроллер карты
  обрабатывает wear-leveling, ECC и переназначение. Мы отказались от сырого NOR из-за риска
  блокировки некоторых чипов при частичных записях и из-за зависимости от производителя.
  Симулятор NOR остаётся только как доказательство универсальности формата
  (ядро независимо от носителя через HAL), а не как поддерживаемый путь.

## 2. Раскладка файла

Мы сохраняем проверенную физическую модель — **два потока, растущие друг навстречу другу**
в файле фиксированного размера:

```
смещение 0                                                            EOF
┌────────────┬───────────────┬──────────┬─────────────┬──────────────┐
│ ПРЕФИКС     │ ДАННЫЕ поток →│ своб. 0xFF│ ← поток ЛОГИ  │ ЗЕРКАЛО ПРЕФ.│
│ 1–255 сект  │ (растёт вверх)│           │ (растёт вниз) │ (копия)      │
└────────────┴───────────────┴──────────┴─────────────┴──────────────┘
             ▲ data_base      ▲ top_ptr  bot_ptr ▲ region_end
```

- **ДАННЫЕ** растут вверх от `data_base` (`top_ptr` = куда пойдёт следующий блок).
- **ЛОГИ** растут вниз от `region_end` (`bot_ptr`; следующая запись идёт
  на `bot_ptr − log_rec_size`).
- Между ними свободное место, заполненное `0xFF`.
- **ПРЕФИКС** по умолчанию один сектор 512 Б; растёт (целыми секторами, до 255)
  только когда таблицы SCHEMA + STATION не помещаются. `data_base` = размер
  префикса.
- **ЗЕРКАЛО ПРЕФИКСА** (добавлено в v1.1): бинарно идентичная копия префикса в
  последних байтах файла (размером с префикс), так что единственный сбойный
  сектор на смещении 0 не ослепляет весь файл — `mount()` откатывается к нему.
  Поэтому поток ЛОГОВ заканчивается на `region_end = file_size − размер префикса`,
  прямо под зеркалом. Отдельного индексного региона нет — на диске нет дерева
  поиска.

### Почему фиксированный, предварительно выделенный размер

Файл **предварительно выделяется полностью** во время `format()` и заполняется `0xFF` 
(точно как современный `MlaPosixHAL.create`). На FAT/SD это правильный выбор:

- цепь кластеров FAT выделяется заранее → файл не растёт, не фрагментируется,
- все **логические смещения стабильны** на протяжении всей жизни файла,
- модель «два указателя друг против друга» подходит для фиксированного региона без конфликтов с ФС.

### Условие «полно»

```
top_ptr + размер_следующего_блока  >  bot_ptr − log_rec_size
```

### Режимы заполнения (`container_kind` в префиксе)

| Значение | Режим | Поведение | Рекомендация |
|---|---|---|---|
| 0 | **Жёсткая остановка** | RuntimeError при переполнении | просто |
| 1 | **Ротация файлов** | при переполнении открывается следующий файл `NIC0001.MLA`, `NIC0002.MLA`, … ; каждый префикс содержит `file_seq` | **рекомендуется для FAT/SD** |
| 2 | **Круговой буфер** | ДАННЫЕ оборачиваются обратно, самый старый сектор освобождается (`sector_erase`) и соответствующие слоты логов помечаются как заброшенные | только RAW/NOR / эксперимент |

Ротация предпочтительна: каждый файл независимо монтируется и crash-safe,
карты огромны, поэтому переполнение редко. Круговой буфер усложняет восстановление и
откладывается (см. §9).

### Решение: размер контейнера и предварительное выделение

- **Свободное место = `0xFF`** (как свежий NOR после стирания) остаётся — **без суперблока**.
  Это проще для МКУ: чип только пишет, `mount()` находит границу сканированием `0xFF`.
  Никакой персистентности указателей в префиксе.
- **Размер контейнера по умолчанию ~1 МБ.** Предварительное выделение 1 МБ (заполнение `0xFF`)
  быстро выполняется один раз на МКУ; для больших объёмов **ротируется несколько 1МБ файлов**.
- **Большие файловые системы** = много 1МБ файлов на огромной карте. **Агрегация и чтение между файлами
  делает ПК** (`MlaArchive`) — мощный процессор «съедает всё», поэтому медленное предварительное
  выделение и полный сканирование — не проблема. МКУ держит открытым только один 1МБ файл.
- 32-битная адресация → один файл максимум **4 ГБ**; свыше того (и ниже) — ротация.
  Если количество файлов вас беспокоит, можно поднять `file_size` (например, 16–64 МБ) —
  это просто выбор в `format()`.

## 3. Префикс (1–255 секторов по 512 Б)

Префикс — это 34 Б структурированный заголовок, за ним две самоописывающие
таблицы (SCHEMA + STATION), и заканчивается CRC16 над всем, что перед ним.
Обычно это **один сектор 512 Б**; если таблицы не помещаются, он растёт целыми
секторами 512 Б (до **255 ≈ 127 КБ**), и CRC переезжает в последние 2 байта.

> Лимит 255 секторов — это жёсткий потолок, а не цель: он существует лишь
> потому, что счётчик однобайтовый. **Рекомендуемый максимум — 16 секторов
> (8 КБ)**; с автоматически рассчитываемыми таблицами SCHEMA/STATION реальная
> станция к нему никогда не приближается.

```
[0]   magic[4]        b"MLA\0"
[4]   version         1 Б   = 1
[5]   cluster_shift   1 Б   8=256B · 10=1KB · 12=4KB · … · 15=32KB
[6]   log_rec_size    1 Б   = 16
[7]   flags           1 Б   режим CRC (биты 0-1): 0=NONE · 1=DATA · 2=FULL
[8]   file_size       4 Б   uint32 LE
[12]  reserved        8 Б   0
[20]  container_kind  1 Б   0=single · 1=rotation
[21]  file_seq        2 Б   uint16 LE  порядок файла в ротации
[23]  keyframe_intv   1 Б   интервал keyframe для сжатия (по умолчанию 0 = N/A)
[24]  enc_caps        1 Б   битовая маска кодирований
[25]  data_base       4 Б   uint32 LE  = размер префикса (первый байт ДАННЫХ)
[29]  region_end      4 Б   uint32 LE  = file_size − размер префикса (конец потока
                                         ЛОГОВ; за ним следует зеркальная копия префикса)
[33]  reserved        1 Б   0
[34]  таблица SCHEMA  …     §3.1
[..]  таблица STATION …     §3.2
[конец-2] mla_crc16       2 Б   LE  — над всем, что перед ним
```

### 3.1 Таблица SCHEMA — имена/единицы полей для CSV/SQL

Строится/читается через `tools/mla_schema.py`. Позволяет любому читателю
экспортировать записи в CSV/SQL **без предварительного знания** — станция несёт
собственное описание столбцов.

```
[0] tbl_ver  1 Б  = 1
[1] n_log    1 Б  число LOG-полей (описывают timestamp и т.п.)
[2] n_data   1 Б  число DATA-полей (столбцы упакованного payload)
[3 ..]       (n_log + n_data) × 14 Б дескрипторов поля:
   width 1 Б · unit 1 Б · exp10 1 Б (i8) · flags 1 Б (бит0=signed) ·
   offset 2 Б (i16 LE) · name 8 Б (UTF-8, дополнено NUL)
   физическое = (raw + offset) × 10^exp10
```

Словарь единиц универсален (общий для спецификации); только *состав* полей
(какие датчики, масштаб, ширина, **8-символьное имя**) специфичен для устройства
и путешествует в файле.

### 3.2 Таблица STATION — индекс → реальная станция

```
[0] sta_ver  1 Б  = 0x53
[1] n        1 Б  число станций (1..255)
[2 ..]       n × 6 сырых байт (индекс i в логе → запись i-1)
```

Эти 6 байт **непрозрачны для MLA**. Частое деление — `region(2) + номер(2) +
reserved(2)`, но решает клей (glue); может быть и `город/номер/край`, или одно
большое число. Люди присваивают номера станций с пропусками — клей отображает их
в компактные 1-байтовые индексы и обратно.

> **Тупой контейнер.** Обе таблицы записываются дословно сверху и путь C/MCU их
> никогда не читает. Сжатие, шифрование, перевод номеров станций и транспорт
> живут в отдельном слое-клее.

## 4. Флаг сжатия (без реестра типов)

### 4.1 Один бит, а не байт типа

v1.1 намеренно **убрал** старый байт `rec_type`/класса (measure / event /
config / delta / keyframe / …). Файлы **однородны** — что *означает* полезная
нагрузка (какой байт температура, какой влажность) задаёт **таблица SCHEMA**
(§3.1), а не тег типа на каждую запись. Тег типа, зависящий от клея, гниёт;
SCHEMA — нет.

Вместо него запись ЛОГОВ несёт лишь байт `flags` (§5):

- **бит 7 — `compressed`**: полезная нагрузка — блоб кодека (например, NIC-DMD).
  Контейнер никогда не заглядывает внутрь; этот бит означает только «при чтении
  передай нагрузку слою кодека».
- **биты 0–6 — `kf_back`**: для сжатого потока — на сколько записей назад лежит
  свой keyframe (0 = эта запись **и есть** keyframe). Полные 7 бит → 0..127,
  поэтому интервал keyframe — выбор писателя/устройства, и читателю его никогда
  не нужно сообщать (keyframe = `kf_back == 0`).

### 4.2 Какой кодек — в блоке данных, а не в записи

Формат **переносит** бит сжатия, но **не определяет метод**. КАКОЙ кодек /
keyframe / вариант — закодировано в **собственном заголовке блока данных**
(NIC-DMD уже так делает: байт 0 блоба). Поэтому добавить новый кодек — это пара
строк в клее, а контейнер не меняется. Два уровня:

- **запись ЛОГОВ** — это индекс: найти/отфильтровать (в т.ч. по биту
  `compressed`) без чтения данных;
- **блок ДАННЫХ** — самоописывающий, непрозрачный блоб:
  `MAGIC · [заголовок кодека 1–4 Б][сжатые данные] · CRC`.

`keyframe_intv` (байт префикса 23) — только подсказка для читателя — каденция,
которую писатель *намеревается* выдержать — и по умолчанию 0. Она никогда не
авторитетна; авторитетен `kf_back == 0`.

**Практический пример:**
1. Клей кодирует: образец 0 = keyframe (сырая опора), образцы 1..6 = дельты,
   образец 7 = снова keyframe (каденция DMD по умолчанию — 7).
2. Контейнер хранит каждый блоб с `compressed = 1` и `kf_back` = расстояние до
   keyframe (0 на keyframe) — никакой интерпретации, только индексация.
3. При чтении декомпрессор (не ядро) воспроизводит от ближайшей записи с
   `kf_back == 0`.

### 4.3 Возможности кодирования (`enc_caps`)

Байт префикса 24 (`enc_caps`) — это рудиментарная, обращённая к читателю
**подсказка**-битмаска, оставленная для прямой совместимости; v1.1 оставляет её
**0** по умолчанию. Она никогда не ограничение — авторитетная информация о
кодировании в собственном заголовке каждого блока данных, поэтому читатель
всегда должен обрабатывать блок на его собственных условиях.

## 5. Запись лога (16 Б)

Запись лога живёт в потоке ЛОГОВ (растёт вниз от `region_end`). Это фиксированные
**16 байт**, и **вся запись покрыта CRC** — никакого поля «flags вне CRC».

```
[0]  offset     4 Б  u32   логическое смещение блока данных в ДАННЫХ
[4]  timestamp  4 Б  u32   Unix-секунды (от RTC/GPS вызывающего)
[8]  subsec     2 Б  u16   два непрозрачных байта (0..65535); смысл задаёт клей, MLA — никакого
[10] length     2 Б  u16   размер payload 1..65535
[12] flags      1 Б  u8    бит7 = compressed, биты0-6 = kf_back (записей назад
                           до своего keyframe; 0 = эта запись И ЕСТЬ keyframe)
[13] station    1 Б  u8    индекс 1..255 в таблицу станций префикса (0 = нет)
[14] mla_crc16      2 Б        CRC16 над [0..13]
```

Почему 16 Б: это степень двойки, поэтому запись никогда не пересекает сектор
512 Б, а адресация слота — это сдвиг, а не умножение — самый дружелюбный размер
для МК. Поле `subsec` (добавлено в v1.1) — это два непрозрачных байта, которые
контейнер лишь несёт: MLA не придаёт им никакого смысла. Имя намеренно читается
двояко: sub-**sec**ond (подсекундное) время *и* sub-**sec**tion (например, индекс
секции / ротации). Смысл задаёт слой-клей — он может использовать поле как одно
16-битное значение, как два независимых байта или сразу для нескольких вещей
(например, старший байт = секция/ротация, младший байт = подсекундный тик для
дискретизации заметно выше 1 Гц). Если не используется — ставьте 0. Единственный
байт `flags` упаковывает бит `compressed` и расстояние `kf_back` (см. §4).

### 5.1 Состояния записи (без поля flags)

Слот интерпретируется чисто по его байтам:

| Состояние | Байты | Определяется по |
|---|---|---|
| **Свободный** | все `0xFF` | свежий / стёртый носитель |
| **Живой** | данные + совпадающий CRC | `mla_crc16(тело) == сохранённый CRC` |
| **Заброшенный** | все `0x00` | CRC не сходится (обнулённое тело **не** хешируется в `0x0000`) |

Отмена записи = **перезаписать 16 Б нулями**. Её CRC тогда не совпадает, и любой
читатель её пропускает. Это заменяет старый трюк «переключить один байт flags вне
CRC» и позволяет покрыть контрольной суммой всю запись.

### 5.2 `station` — это индекс, а не номер

`station` — **1-байтовый индекс** (1..255; 0 = нет) в таблицу STATION в префиксе
(§3.2). Реальные номера станции/региона — которые люди и инструменты назначают
как угодно, с пропусками — живут в этой таблице; контейнер индекс никогда не
интерпретирует. Перевод индекс ↔ реальный номер — задача хостового клея.

> Никаких контрольных точек. Размер файла фиксирован, а лог имеет фиксированный
> шаг, поэтому `mount()` находит границу двоичным поиском и читает `offset +
> length` новейшей валидной записи, чтобы восстановить `top_ptr` — сохранять
> нечего, поэтому записи контрольной точки не существует.

## 6. Блок данных (переменный)

Полезная нагрузка, записанная в поток ДАННЫХ:

```
[0]       magic       2 Б  0xAB 0xCD  (синхрослово)
[2]       <payload>   N Б  данные приложения (1 до 65535 Б)
[2+N]     mla_crc16       2 Б  LE  — CRC16 над payload (0xFFFF, если режим CRC = NONE)
```

**Почему нет байта типа в блоке?** В v1.1 типа на запись нет вообще — файлы
однородны, а смысл задаёт SCHEMA, а не тег. Это держит поток данных **чисто
управляемым приложением**. Со схемой в префиксе payload — это столбцы датчиков,
упакованные подряд (§3.1); `mla_decode_payload()` делит и масштабирует их в
`(имя, единица, значение)` для CSV/SQL.

## 7. Crash-безопасность

Протокол — **LOCK первым, DATA вторым**:

1. **Оборванная запись лока** (прервано во время записи LOG-записи) → у этого
   слота плохой CRC → пропускается при монтировании. Двоичный поиск границы
   продолжается.
2. **Оборванная запись данных** (LOG OK, но блок данных неполон) → отсутствует
   `MAGIC` → при монтировании лок **обнуляется** (все 16 Б перезаписаны `0x00`,
   так что его CRC не сходится) и `top_ptr` возвращается к `rec.offset`.
3. **Отмена** любой записи делается так же — перезаписать нулями; CRC тогда не
   сходится, и читатели её пропускают. Нет байта flags и ничего вне CRC.
4. `recover()`: находит `MAGIC`, пробует длины 1..65535, пока `CRC16(payload)`
   не совпадёт; восстановленные записи помечаются несжатыми (`compressed = 0`,
   `kf_back = 0`), поскольку кодирование из самого блока восстановить нельзя.
5. Никаких контрольных точек: размер файла фиксирован, лог с фиксированным шагом,
   поэтому двоичный поиск + чтение новейшей записи восстанавливают состояние
   напрямую. Скан можно начать грубым шагом (например, 256, для большого файла
   2000) и при попадании на `0xFF` отступить на один — на диске нечего чинить.

## 8. Настраиваемые параметры (устанавливаются в `format()`, хранятся в префиксе)

| Параметр | Где | Варианты | По умолчанию |
|---|---|---|---|
| `cluster_shift` | байт 5 | 8…15 (256 Б … 32 КБ) | 12 (4 КБ) |
| `flags` (CRC) | байт 7, биты 0-1 | NONE / DATA / FULL | FULL |
| `container_kind` | байт 20 | single / rotation | single |
| `file_seq` | байт 21 | 0…65535 | 0 |
| `keyframe_intv` | байт 23 | 0…255 | 0 |
| `enc_caps` | байт 24 | битовая маска кодирований | по применению |
| `schema_table` | [34..) | из `tools/mla_schema.py` | пусто |
| `station_table` | после схемы | из `tools/mla_schema.py` | пусто |

`log_rec_size` фиксирован на **16**, а `data_base` вычисляется (= размер
префикса, который равен 512 Б, если таблицы не выходят за пределы сектора).

## 9. Вне области применения (живёт в слое-клее, не в MLA)

MLA — тупой контейнер; следующее намеренно **не** его работа:

- **Перевод номеров станций** — лог хранит 1-байтовый индекс; отображение его в
  реальный, возможно дырявый номер станции — задача клея (через таблицу STATION,
  которую он записал).
- **Сжатие** — MLA лишь помечает его (бит `compressed`; `kf_back` связывает
  запись с её keyframe, 0 = keyframe). Кодек, и какой вариант, отдельный и живёт
  в собственном заголовке блока данных.
- **Шифрование** — то же: отдельная библиотека; MLA хранит любые полученные байты.
- **Транспорт (LoRa / Wi-Fi / сеть)** — каждая запись самодостаточна (тип +
  длина + CRC), так что «отправить запись» = отправить её байты. Выбор
  транспорта — за клеем.
- **Ротация файлов** по многим файлам — платформенный клей над файловой системой
  (`MlaArchive` в Python); каждый файл монтируется отдельно через `file_seq`.

Это разделение держит MLA достаточно малым для ATmega (только запись, 16 Б лог,
один сектор префикса 512 Б), позволяя при этом мощному хосту строить над ним
сколь угодно умную систему.
